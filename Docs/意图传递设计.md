# 意图传递设计

## 问题

```
找到 2F 的 job A → UseStairs 去 2F → ThinkTree 重新评估 → 原版随机选了 job B ❌
```

pawn 专门为 A 跑过去，到了之后原版重新扫描，可能选了离楼梯很远的 B，白走一大段。
更糟的情况：原版在 2F 什么都没找到（条件变了），pawn 发呆，下一 tick 又被送回 1F。

## 解决方案

在 UseStairs 之外加一层"意图"，记住 pawn 为什么跨层，到达后直接执行。

```
找到 2F 的 job A → 保存意图(A) → UseStairs 去 2F → 读取意图 → 执行 A ✅
                                                    ↓ A 失效
                                              清除意图 → 原版接管
```

## 意图 vs 直接塞 Job

不能在出发前就创建完整 Job，因为：
- `Reserve` 要求 pawn 和 Thing 同图
- `JobOnThing` 内部可能检查 pawn 位置
- 跨层途中目标可能消失/被占

意图只记"去哪、找什么"，到达后再创建 Job。

## 数据结构

```
Intent:
  destMapId     — 目标地图 ID
  targetPos     — 目标 Thing 的位置（IntVec3）
  targetDef     — 目标 Thing 的 ThingDef
  createdTick   — 创建时间（用于过期）
```

按 pawnId 索引，静态字典。不存 Job 对象，不存 Thing 引用（跨图引用不安全）。

存档：不需要。UseStairs 是短时 job，存档重载后 ThinkTree 自然重新评估。

## 意图的生命周期

```
创建：JobGiver postfix 找到跨层目标时
  ↓
携带：pawn 执行 UseStairs，意图静静等着
  ↓
消费：transfer 后，UseStairs 的续接 toil 读取意图
  ↓ 成功
创建 Job → StartJob → 清除意图
  ↓ 失败（Thing 没了 / 不可达 / 被占）
清除意图 → UseStairs 正常结束 → ThinkTree 接管
```

过期：超过 2500 tick（~1 分钟）自动清除，防止僵尸意图。

## 到达后怎么创建 Job

### 需求类（食物/休息/娱乐）

不需要意图。原因：需求是可替代的。

- 食物：2F 有 10 个食物，吃哪个都行
- 床：pawn 有自己的床，原版直接找到
- 娱乐：任何娱乐设施都行

pawn 到达 2F 后，原版 GetFood/GetRest/GetJoy 自然在本层找到资源。
不需要精确到"吃第 3 个简单食物"。

### 工作类（建造/搬运/清洁/...）

需要意图。原因：工作是特定的。

pawn 为了建造 2F 的墙壁 A 跨层，到了之后应该建墙壁 A，不是去扫地。

到达后的执行流程：

```
1. 在当前地图的 targetPos 附近找 targetDef 的 Thing
2. 找到 → 遍历 pawn 的 WorkGiver 列表（按优先级）
3. 对每个 WorkGiver 调用 HasJobOnThing(pawn, thing)
4. 第一个返回 true 的 → JobOnThing(pawn, thing) → StartJob
5. 全部失败 → 清除意图，原版接管
```

为什么遍历 WorkGiver 而不是存 WorkGiverDef：
- 简单，不需要在意图里多存一个字段
- 同一个 Thing 可能被多个 WorkGiver 处理（建造 vs 修复）
- pawn 的工作设置可能在途中改变

## UseStairs 的变化

```
现在：
  Toil 1: 走到楼梯
  Toil 2: Transfer

改后：
  Toil 1: 走到楼梯
  Toil 2: Transfer
  Toil 3: 读取意图 → 尝试创建 Job → StartJob 或清除
```

Toil 3 是 Instant toil，不阻塞。成功则 pawn 无缝切换到目标 job，失败则 UseStairs 正常结束。

## 搬运类的特殊处理

P1（本层材料 → 其他层蓝图）和 P2（其他层材料 → 本层蓝图）已经有自己的 job 类型（HaulToStairs / HaulAcrossLevel），不走意图系统。

意图只服务于 P3（去其他层找工作）和未来可能的扩展。

## 边界情况

### 目标消失
Thing 被其他 pawn 拿走/建完/摧毁 → 意图执行失败 → 清除 → 原版接管。
pawn 已经在 2F 了，原版会在 2F 找其他工作，不会白跑。

### 多 pawn 同目标
两个 pawn 都为同一个 Thing 跨层。先到的成功 Reserve，后到的意图执行失败。
后到的 pawn 在 2F 找其他工作。正常行为。

### 途中优先级变化
pawn 在走楼梯途中饿了 → ThinkTree 中断 UseStairs → 意图过期自动清除。
吃完后 ThinkTree 重新评估，可能重新跨层（创建新意图），也可能在本层找到工作。

### 楼梯被拆
UseStairs 已经有 `FailOnDespawnedOrNull(TargetIndex.A)`。
楼梯被拆 → job 失败 → 意图清除。

## 不做的事

1. ❌ 不存完整 Job 对象 — 跨图 Job 的 target 引用不安全
2. ❌ 不存 Thing 引用 — 跨图 Thing 引用可能野指针
3. ❌ 不给需求类加意图 — 需求可替代，原版处理更好
4. ❌ 不持久化意图 — UseStairs 是短时 job，存档后重新评估
5. ❌ 不在意图里存 WorkGiverDef — 到达后遍历更灵活
